Source: `src/lib/alert-engine.ts` — `runHealthCheck()`, `sendAlertEmails()`  
Source: `src/lib/health.ts` — inline alert generation in `calculateHealth()`

---

## Alert Levels

| Level | Color | Meaning |
|-------|-------|---------|
| `critical` | Red | Immediate action required. Emailed to org members. |
| `warning` | Amber | Action required this sprint. Not emailed by default. |
| `info` | Blue | Informational. No action required. |
| `success` | Green | All checks passing. |

---

## Alert Categories

| Category | Covers |
|----------|--------|
| `schedule` | SPI deviations, overdue, sprint misalignment |
| `budget` | CPI deviations, EAC vs BAC, TCPI |
| `scope` | Blocked features, backlog growth, missing modules |
| `risk` | Open high risks, unmitigated threats |
| `resources` | Team overload, underutilisation |
| `progress` | General progress alerts, data quality |
| `governance` | Missing documents, unapproved items |

## Alert Conditions (Alert Engine Cron)

These run via `runHealthCheck()` for all non-CLOSED/ARCHIVED/COMPLETED projects:

### 1. Project Overdue
- **Trigger**: `daysLeft < 0` AND `progressNominal < 100`
- **Level**: critical
- **Dedup**: 24h
- **Message**: "Deadline was N days ago with X% work remaining"

### 2. Critical SPI
- **Trigger**: `SPI < 0.5`
- **Level**: critical
- **Dedup**: 24h
- **Message**: EVM deviation, % complete vs planned

### 3. Warning SPI
- **Trigger**: `0.5 ≤ SPI < 0.8`
- **Level**: warning
- **Dedup**: 48h (less urgent)
- **Message**: Behind schedule, schedule variance

### 4. Critical Budget
- **Trigger**: `CPI < 0.7` AND `budgetTotal > 0`
- **Level**: critical
- **Dedup**: 24h
- **Message**: EAC vs BAC, instructions to freeze spending

### 5. Team Blocked
- **Trigger**: `blockedFeatures >= 3`
- **Level**: critical
- **Dedup**: 12h (shorter — blocking is acute)
- **Message**: Count of blocked features, velocity impact

### 6. Sprint Milestone Risk
- **Trigger**: active sprint ending in ≤ 2 days with `sprintProgress < 80%`
- **Level**: warning
- **Dedup**: 24h per sprint
- **Message**: Sprint name, days remaining, completion %

## Alert Conditions (Health Engine — inline)

Generated by `calculateHealth()` and returned in `HealthReport.alerts`. These are not persisted — they're computed on-demand for the UI:

| Alert ID | Trigger | Level |
|----------|---------|-------|
| `overdue` | daysLeft < 0 and progress < 100% | critical |
| `spi-critical` | SPI < 0.5 | critical |
| `spi-warning` | 0.5 ≤ SPI < 0.8 | warning |
| `cpi-critical` | CPI < 0.7 and BAC > 0 | critical |
| `cpi-warning` | 0.7 ≤ CPI < 0.9 and BAC > 0 | warning |
| `tcpi` | TCPI > 1.2 and BAC > 0 | warning |
| `blocked-critical` | blockedFeatures ≥ 3 | critical |
| `blocked-warning` | 0 < blockedFeatures < 3 | warning |
| `time-progress` | forecaseMode = time_vs_progress AND scheduleGap < -30 | critical |
| `high-risks` | highRisks > 0 | warning |
| `overloaded` | utilization > 100% | warning |
| `insufficient-data` | forecastMode = insufficient | info |
| `healthy` | no other alerts | success |

## Deduplication Rules

- Alerts are stored in the `Alert` table with `createdAt` timestamp
- Before creating an alert, `wasRecentlySent(projectId, type, hours)` checks for existing records
- Default window: **24h**
- Shorter windows: `team_blocked` = 12h (acute), `spi_warning` = 48h (not urgent)
- Sprint milestone alerts are deduped per sprint ID (`milestone_${sprint.id}`)

## Email Delivery

`sendAlertEmails(organisationId, userEmail)`:
- Fetches all `critical` alerts where `emailSent = false`
- Sends one digest email with up to 10 alerts
- Uses Resend with `noreply@roadmapai.com`
- Marks alerts `emailSent = true` after confirmed delivery
- Only critical alerts are emailed — warnings and info are UI-only

---

## Alert Lifecycle

```
Health check runs → conditions evaluated
  │
  ├─ wasRecentlySent? → skip
  │
  └─ create Alert record (emailSent: false)
       │
       └─ sendAlertEmails() called (separately)
            └─ Resend delivers digest → emailSent: true
```

Alerts are never automatically deleted. The UI shows the last 50 alerts. Users can mark individual alerts as read.

## Guardian AI Alert vs Engine Alert

| Type | Source | Persisted | Emailed |
|------|--------|-----------|---------|
| Engine alert | `runHealthCheck()` | Yes — Alert table | Yes (critical only) |
| Health inline alert | `calculateHealth()` → HealthReport.alerts | No — in-memory | No |
| Guardian alert | `runGuardianAgent()` → GuardianReport | Yes — GuardianReport JSON | No |

Guardian alerts are richer (narrative, category, action) but not emailed. Engine alerts are leaner but trigger emails.
