# RoadmapAI — Functional Analysis

**Version:** 2.0  
**Date:** April 2026  
**Type:** AI Decision Layer for Project Portfolio Management

---

## 1. Executive Summary

RoadmapAI is a multi-tenant AI-powered platform that sits above traditional project management tools as a **decision layer** — transforming raw project data into clear, actionable decisions for PMs, PMO leads, and executives.

While tools like Jira, Asana, and Monday manage task execution, RoadmapAI answers the questions they cannot:

- Is this project actually on track, or just appearing to be?
- Where will we lose money before it happens?
- What is the single most important thing to do right now?

The platform is built on PMI/PMBOK Earned Value Management (EVM) standards for quantitative health scoring, with Guardian — an AI agent powered by Anthropic Claude — embedded throughout every view to surface context-specific recommendations, predictions, and decision prompts.

**Target users:**
- Project Managers who need to spot problems early and act fast
- PMO Leads governing portfolios of 3–50 projects with financial accountability
- Executives (CEO/CFO) who need a single-number health signal and budget risk visibility without navigating dashboards

**Competitive positioning:**
RoadmapAI does not compete with Jira or Monday on task management. It competes on decision intelligence — the layer above execution tools that answers *what to do*, not just *what is happening*.

## 2. User Roles and Permissions

The system operates on a three-tier role model scoped to each Organisation. All permissions are enforced at both the API and UI layer.

| Role | Description | Key Capabilities |
|---|---|---|
| **ADMIN** | Full control of the organisation | All actions: create/edit/delete projects, manage team, manage billing, edit resources, share projects |
| **MANAGER** | Operational project lead | Create/edit projects, manage features and sprints, edit risks, manage dependencies, invite members |
| **VIEWER** | Read-only stakeholder | View all project data including financials, risks, and Guardian reports — no write access |

Permission boundaries enforced:
- Only ADMINs can delete projects, manage billing, edit resources, and remove members
- Only ADMINs and MANAGERs can create projects, edit features, edit risks, and manage sprints
- Financial data (`viewFinancials`) is accessible to all authenticated members
- Share links grant unauthenticated read access to a specific project

## 3. Core Functional Modules

### 3.1 Organisation & Multi-Tenancy

RoadmapAI is fully multi-tenant. Each Organisation is a self-contained workspace:

- Provisioned automatically on first login via Clerk organisation sync
- Contains all projects, members, departments, resources, and billing configuration
- Data is strictly isolated — no cross-organisation data access is possible
- Supports custom departments with budget tracking and colour coding

**Onboarding flow:** New users who have no projects are redirected to an onboarding wizard that walks through organisation setup, first project creation, and team invitation.

---

### 3.2 Project Lifecycle Management

A **Project** is the central entity. It passes through a defined lifecycle:

```
NOT_STARTED → ACTIVE → PAUSED → COMPLETED → CLOSED
                                           → ARCHIVED
```

Every status transition is:
- Logged in `ProjectStatusLog` with timestamp, actor, and optional note
- Recorded in the `Activity` audit trail
- Visible in the Governance view's status history

**Project fields:**
- Name, description, functional analysis (AI-generated on creation)
- Start date and end date (used for EVM schedule calculations)
- Budget total (BAC — Budget at Completion)
- Revenue expected
- Health score (computed, stored as cached value)
- Requested by (optional stakeholder link to a team member)
- Department assignments (many-to-many)

**Project closure:** When a project moves to CLOSED status, a closure report is automatically generated by Claude AI in the background, summarising final performance, lessons learned, and delivery assessment.

**Project sharing:** Admins and Managers can generate a public share token that grants read-only access to the project without authentication. Useful for client or executive stakeholder access.

---

### 3.3 Project Structure: Phases, Sprints, Features

Projects are broken down hierarchically:

```
Project
  └── Phase (numbered milestone groups, e.g., "Phase 1: Discovery")
        └── Sprint (time-boxed delivery units with start/end dates)
              └── Feature (atomic work items)
```

**Phases:**
- Represent high-level delivery milestones
- Have a label, subtitle, colour accent, and sequential order
- Progress through phases is inferred from sprint completion ratios

**Sprints:**
- Have a status: UPCOMING → ACTIVE → DONE
- Status auto-advances: when any feature moves to IN_PROGRESS, the sprint becomes ACTIVE; when all features are DONE, sprint becomes DONE
- Linked optionally to a phase for roadmap grouping
- Sprint velocity is tracked and used in EVM delay forecasting

**Features (work items):**
- Status: TODO / IN_PROGRESS / DONE / BLOCKED
- Priority: CRITICAL / HIGH / MEDIUM / LOW
- Estimated hours and actual hours for cost tracking
- Assignable to a Resource (team member/contractor)
- Module field for logical grouping within a sprint
- Notes field for free-text context
- Feature-to-feature dependency tracking (blocker/blocked-by graph)

**Auto-progression logic:**
- Feature status change → sprint status recalculated automatically
- Sprint status change → project status recalculated (all sprints DONE → project COMPLETED)

### 3.4 Project Views — Decision-Oriented Navigation

The project detail page is structured around **what the user needs to decide**, not around internal system functions. Views are reduced from 7 technical tabs to **5 decision-oriented views**, with Guardian embedded as a live AI layer inside each one rather than isolated in a separate tab.

**Design principle:** Every view answers one specific question. The user navigates between decisions, not between features.

---

#### View 1: Health & Decisions *(primary view — opens by default)*

**Question answered:** "Is this project in danger, and what do I do right now?"

**Target users:** PM, PMO Lead, Executive

This is the most important view in the product — the entry point for every project session and the primary source of decision prompts.

**Content:**

*Status strip (top of view):*
- RAG status badge: ON TRACK / AT RISK / CRITICAL (colour-coded)
- Health score (0–100, EVM-based)
- On-track probability (0–100%)
- Delay prediction: "+12 days estimated delay"
- Budget risk: "+18% forecast overrun"

*Decision Feed (central differentiator):*

The Decision Feed is a live, prioritised list of AI-generated action prompts. Each item is a concrete decision the PM can take today, with quantified impact:

```
[CRITICAL]  Scope phase 2 down by 3 features
            → saves 8 days · removes critical path blocker
            [Apply]  [Ignore]

[WARNING]   API dependency will block Sprint 4 in 6 days
            → assign owner now to resolve before impact
            [Assign owner]  [Dismiss]

[INFO]      Team utilisation at 108% — burnout risk
            → redistribute 12h from Dev to QA
            [View breakdown]
```

Each decision item includes: priority level, plain-language description, quantified consequence, and a primary action button.

*Supporting panels:*
- EVM metrics strip (SPI, CPI, EAC — simplified, not raw numbers)
- Guardian narrative: 2–3 sentence AI explanation of why the health score is what it is
- Top 3 active alerts from the alert engine

**Guardian layer:** Fully embedded. The Decision Feed is generated by Guardian using EVM data + agentic tool analysis. Guardian does not appear as a separate tab — it is the intelligence engine behind this view.

#### View 2: Execution

**Question answered:** "What is the team working on right now, and what is blocked?"

**Target users:** Developer, PM

**Content:**

*Board sub-view (default):*
- Kanban columns: TODO / IN_PROGRESS / DONE / BLOCKED
- Sprint selector (active sprint shown by default)
- Feature cards with priority badge, assignee, estimated hours
- Drag-and-drop status updates

*Backlog sub-view:*
- Full feature list across all sprints
- Filters: by sprint, status, priority, assignee, module
- Inline editing of title, priority, hours, assignee
- Feature-to-feature dependency indicators (blocked-by chain)

**Guardian layer:** Blocked features surface AI-generated unblocking suggestions inline on the card ("This feature is blocked by X — suggested action: Y").

---

#### View 3: Plan & Roadmap

**Question answered:** "Are we on the right timeline, and how do phases align with delivery?"

**Target users:** PM, Executive, Client (via share link)

**Content:**
- Horizontal timeline/Gantt showing phases and sprints across calendar
- Sprint start/end dates with completion percentage overlaid
- Phase milestone markers
- Forecast end date shown alongside planned end date (delta highlighted when diverging)
- Dependencies between sprints visualised as connectors

**Guardian layer:** Delay forecast banner at top — "At current velocity, delivery expected [date] — [N] days after planned deadline." Includes a one-click prompt to open the Decision Feed for remediation options.

#### View 4: Financial Intelligence

**Question answered:** "Where are we losing money, how much will we lose, and what do we do about it?"

**Target users:** Finance, CFO, PM

This view goes beyond displaying numbers — it interprets them and prescribes actions.

**Content:**

*Financial health panel:*
- Health score breakdown (Schedule 35% / Cost 30% / Resource 20% / Risk 15%)
- Budget: planned vs actual vs EAC forecast (with variance bar)
- Revenue expected vs margin
- Daily burn rate vs planned burn rate

*Cost intelligence cards:*
- "You are on track to overspend by $[X]" — with root cause (which resource / sprint)
- "At current CPI ([value]), you need to perform [X]% more efficiently to finish on budget (TCPI)"
- "Reducing scope by [N] features in Sprint [X] would save ~$[Y]"

*Resource cost breakdown:*
- Per-resource: estimated cost vs actual cost vs forecast
- Team utilisation vs capacity (overload detection)

**Guardian layer:** AI financial recommendations embedded at bottom — prioritised list of cost-saving actions with estimated dollar impact, generated from CPI/TCPI/EAC analysis.

**Edit mode:** ADMIN/MANAGER can edit budget total, actual cost, and revenue expected inline.

---

#### View 5: Risks & Dependencies

**Question answered:** "What can derail this project, and what is blocking us right now?"

**Target users:** PMO Lead, Risk Manager, PM

Focus is deliberately on *current blockers and active threats*, not historical risk logging.

**Content:**

*Active blockers panel (top):*
- Features currently in BLOCKED status with dependency chain
- Cross-project dependencies that could impact this project
- Sprint milestone warnings (sprint ending in ≤2 days with <80% completion)

*Risk register:*
- Risk list ordered by severity (probability × impact score)
- Risk matrix quadrant (probability vs impact heat map)
- Status: OPEN / MITIGATED / CLOSED
- Owner assignment and mitigation notes

*Project dependency graph:*
- Which projects this project depends on
- Which projects are blocked by this project
- Visualised as a linked node graph

**Guardian layer:** For each OPEN high-severity risk, Guardian surfaces an AI-generated mitigation suggestion inline. Accessible via "Suggest mitigation" button per risk row without leaving the view.

---

#### View 6: History & Control *(accessible via secondary navigation)*

**Question answered:** "How has this project changed, and what happened when?"

**Target users:** PMO Lead, Compliance, Executive

**Content:**
- Project snapshot history (version list with reason label and timestamp)
- Snapshot comparison: current state vs selected baseline (scope changes, deadline shifts, budget delta)
- Snapshot restore capability
- Status change log (who changed what status, when, with note)
- Activity audit trail (all user actions chronologically)

**Guardian layer:** When a snapshot comparison shows scope creep or deadline slippage, Guardian annotates the change with an impact assessment ("Scope increased by 8 features since baseline — estimated delivery impact: +14 days").

### 3.5 Financial Engine (EVM)

The financial engine is built on the PMI/PMBOK Earned Value Management standard. All metrics are recalculated in real-time from live project data.

**Core EVM metrics computed:**

| Metric | Full Name | Formula |
|---|---|---|
| BAC | Budget at Completion | Project budget total |
| EV | Earned Value | BAC × (features done / total features) |
| PV | Planned Value | BAC × (days elapsed / total days) |
| AC | Actual Cost | Sum of (actual hours × cost per hour) across all assignments |
| SPI | Schedule Performance Index | EV / PV |
| CPI | Cost Performance Index | EV / AC |
| EAC | Estimate at Completion | BAC / CPI |
| ETC | Estimate to Complete | EAC − AC |
| VAC | Variance at Completion | BAC − EAC |
| SV | Schedule Variance | EV − PV |
| CV | Cost Variance | EV − AC |
| TCPI | To-Complete Performance Index | (BAC − EV) / (BAC − AC) |

**Health Score (0–100):**

A composite PMO health score is calculated from four weighted components:

| Component | Weight | Basis |
|---|---|---|
| Schedule Health | 35% | SPI performance band |
| Cost Health | 30% | CPI performance band |
| Scope Health | 20% | Blocked feature ratio |
| Risk Health | 15% | Maximum risk score (probability × impact) |

Hard caps are applied per PMI RAG thresholds:
- Overdue projects: max score 45
- Severe delay (>30 days): max score 35
- Critical budget overrun: max score 40

**Budget risk levels:** none / low / medium / high / critical (based on EAC vs BAC variance percentage)

**Schedule forecast modes:**
- `velocity` — sprint velocity-based forecast (requires 2+ completed sprints)
- `time_vs_progress` — for short projects with insufficient sprint data
- `overdue` — when deadline has passed with incomplete work
- `insufficient` — insufficient data, monitoring only

**Financials View metrics displayed:**
- Budget, Actual Cost, Forecast (EAC), Revenue Expected, Margin, Daily Burn Rate
- Progress: planned % vs actual % with visual bar comparison
- Sub-scores for Schedule, Cost, Resource, Risk
- AI-generated financial recommendations

### 3.6 Risk Management

The risk register allows tracking of all project risks with a structured probability/impact matrix.

**Risk fields:**
- Title and description
- Probability (1–5 integer scale)
- Impact (1–5 integer scale)
- Risk score = probability × impact (max 25)
- Status: OPEN / MITIGATED / CLOSED
- Mitigation plan (free text)
- Owner name and ID

**Risk categorisation:**
- Categories: schedule, budget, resources, scope, risk, governance (free text)
- Risks are sorted by severity (probability × impact descending)

**AI mitigation suggestions:**
- Available from the Governance view
- Sends risk details to Claude and returns structured mitigation recommendations

**Risk thresholds used in health scoring:**
- Score ≥ 20 → critical risk level
- Score ≥ 12 → high risk level
- Score ≥ 6 → medium risk level
- Score < 6 → low risk level

---

### 3.7 Resource Management

Resources represent people or contractors assigned to projects within an organisation.

**Resource fields:**
- Name, role (free text)
- Cost per hour
- Capacity hours (default 160h/month)
- Department assignment

**Resource assignments (project-level):**
- Each resource can be assigned to a project with estimated hours and actual hours
- Actual cost = sum of (actual hours × cost per hour) across all assignments
- Team utilisation = total actual hours / total capacity hours (as %)
- Overload detected when utilisation > 100%

**Feature-level assignment:**
- Individual features can be assigned to a specific resource
- Enables granular workload tracking

### 3.8 Portfolio Management

The Portfolio view provides an executive-level overview of all active projects in the organisation.

**Portfolio dashboard shows:**
- All active, paused, and not-started projects in a single table
- Per-project: health score, status badge, completion %, days remaining, budget variance, open risks
- Portfolio-level KPIs: total projects, on-track count, at-risk count, total budget, total actual cost
- Health distribution bar (on-track / at-risk / off-track / not started)
- Portfolio-level critical insights (overdue, high-risk, blocked projects)

**Project dependency view:**
- Identifies which projects block other projects
- Visualises the cross-project dependency graph in the Governance view
- Prevents circular dependencies

### 3.9 Guardian AI Agent

Guardian is the AI intelligence layer powered by Anthropic Claude. It is not a standalone feature — it is the engine that powers the Decision Feed, financial recommendations, risk mitigations, delay predictions, and snapshot impact assessments across all five project views.

**Architecture principle:** Guardian runs behind every view and surfaces output where the user already is. The user never needs to "go to Guardian" — Guardian comes to them.

**Two underlying analysis modes:**

1. **Synchronous analysis** (`guardian.ts`):
   - Triggered automatically when a Guardian report is needed and no fresh cache exists
   - Computes all EVM metrics first (pure function, no AI)
   - Sends a structured prompt to Claude with all project KPIs
   - Instructs Claude to return strict JSON: health explanation, alerts (with category, level, action), recommendations, risk level, on-track probability, budget risk, estimated delay
   - Health score is pre-computed by EVM engine and passed in — Claude explains it, never overrides it
   - Falls back to rule-based deterministic analysis if the AI call fails (network error, rate limit, parse failure)

2. **Agentic analysis** (`guardian-agent.ts`):
   - Triggered on manual request or deep analysis
   - Uses Anthropic Tool Use in an agentic loop (up to 5 iterations)
   - Claude decides which tools to call based on what it finds:
     - `get_project_metrics` — full EVM summary
     - `get_sprint_velocity` — velocity trend and acceleration/deceleration signal
     - `get_risk_assessment` — open risk breakdown by category and severity
     - `get_resource_utilisation` — capacity, overload, cost-per-hour breakdown
     - `get_blocked_features` — blocked feature list with dependency context
     - `get_snapshot_comparison` — scope/deadline/budget change detection vs baseline
     - `get_dependency_graph` — feature critical path and active blockers
   - All tool implementations are pure functions over in-memory project data — zero additional DB calls during the agent loop

**Decision Feed generation:**

The Decision Feed (displayed in the Health & Decisions view) is generated by combining:
1. EVM-derived alert conditions (deterministic thresholds — SPI, CPI, blocked count, deadline proximity)
2. Guardian AI recommendations (narrative context, impact quantification, suggested actions)
3. Snapshot comparison deltas (scope creep detection, deadline drift)

Each Decision Feed item is structured as:
```json
{
  "priority": "critical | warning | info",
  "title": "Short action title",
  "detail": "Plain-language explanation with specific numbers",
  "impact": "Quantified consequence (days saved, cost avoided)",
  "action": "Primary CTA label",
  "actionType": "approve | assign | view | dismiss"
}
```

**Guardian report output (cached `GuardianReport` per project):**
- Health score (EVM-validated, not AI-generated)
- Risk level: low / medium / high / critical
- On-track probability (0–100%)
- Estimated delay in days
- 2–5 structured alerts (id, level, category, title, detail, action)
- 3+ concrete recommendations
- Budget risk assessment
- Cached until the next feature update (write-through invalidation)

**Portfolio Guardian:**
- Runs individual Guardian analysis on every active project in the organisation
- Aggregates results into a portfolio-level report
- Surfaces top risks across all projects and cross-portfolio recommendations
- Powers the dashboard's Critical Insights section

### 3.10 Alert Engine

The alert engine runs as a background health check (triggered by cron or manually) and creates structured alerts in the database.

**Alert types generated:**

| Type | Trigger | Level |
|---|---|---|
| `overdue` | Deadline passed with incomplete work | Critical |
| `spi_critical` | SPI < 0.5 | Critical |
| `spi_warning` | 0.5 ≤ SPI < 0.8 | Warning |
| `budget_critical` | CPI < 0.7 and budget is set | Critical |
| `team_blocked` | ≥ 3 features blocked | Critical |
| `milestone_*` | Active sprint ending in ≤ 2 days with < 80% complete | Warning |

**Deduplication:** Alerts of the same type are not re-created within a configurable cooldown window (12–48 hours depending on type).

**Email delivery:**
- Critical alerts are sent via Resend email to the organisation admin
- HTML email template with project links and recommended actions
- Email delivery is tracked (`emailSent` flag) to prevent duplicate sends
- Emails are only marked sent after confirmed API success

### 3.11 Project Snapshots & Version Control

Snapshots capture the full state of a project at a point in time.

**When snapshots are created:**
- Automatically when the project deadline or budget is changed (auto-snapshot via background fetch)
- Manually by users from the Governance view with a custom reason label
- Each snapshot stores: full sprint/feature/phase structure, budget, dates, reason, and version number

**Snapshot comparison:**
- Guardian agentic analysis can compare current state to the most recent snapshot
- Detects scope creep, deadline slippage, and budget changes since baseline

**Snapshot restore:**
- Users can restore a previous snapshot from the Governance view

## 4. User Journeys

### 4.1 New Organisation Setup
1. User signs up via Clerk (email/social auth)
2. Creates or joins an organisation
3. Redirected to onboarding wizard
4. Creates first project — AI generates functional analysis, phases, sprints, and features automatically from a plain-text description
5. Invites team members via email with role assignment
6. Dashboard becomes active with live EVM health scoring from day one

### 4.2 Daily PM Workflow — Decision-First
1. PM opens dashboard → reads Critical Insights (top 5 auto-detected issues across portfolio)
2. Clicks into an at-risk project → **Health & Decisions view opens by default**
3. Reads Decision Feed → sees 3 prioritised action prompts with quantified impact
4. Clicks "Apply" on the top item → scope change is logged, snapshot auto-created
5. Switches to **Execution view** → moves relevant features to reflect scope change
6. Guardian updates Decision Feed automatically after feature changes

### 4.3 Executive Morning Review (5 minutes)
1. Executive opens dashboard → reads 6 KPI cards and portfolio health bar
2. Sees 2 projects flagged AT RISK, 1 CRITICAL
3. Clicks CRITICAL project → **Health & Decisions view**
4. Reads: "Forecast overrun +$42,000 · Delivery +18 days"
5. Reads Decision Feed → sees 2 high-priority decisions with recommended actions
6. Approves one action, delegates the second to the PM via note
7. Leaves project — total time: under 3 minutes

### 4.4 Weekly PMO Governance Review
1. PMO lead opens Portfolio view → reviews all health scores and budget variance at a glance
2. Filters by AT_RISK/OFF_TRACK → identifies 2 projects needing attention
3. Opens first project → **Risks & Dependencies view**
4. Reads active blockers panel — sees one dependency blocking Sprint 4
5. Clicks "Suggest mitigation" on highest-severity risk → Guardian surfaces 3 options inline
6. Assigns mitigation owner and updates risk status to MITIGATED
7. Opens **History & Control view** → runs snapshot comparison to check for scope creep since last month
8. Guardian annotates the diff: "+6 features added, +11 days estimated impact"
9. PM informed, steering committee decision logged as activity

### 4.5 Project Closure
1. PM sets project status to CLOSED
2. System automatically triggers AI closure report generation in the background
3. Status transition logged with actor, timestamp, and optional note
4. Project moves to Archive — closure report available immediately
5. Closure report contains: final EVM metrics, delivery assessment, lessons learned, AI-generated retrospective summary

## 5. Product Positioning & Design Principles

### 5.1 What RoadmapAI Is Not

| Tool | What it does | Why RoadmapAI is different |
|---|---|---|
| Jira / Linear | Task tracking and sprint management | RoadmapAI does not compete on task depth — Execution view is intentionally simple |
| Asana / Monday | Work management and team coordination | RoadmapAI does not manage to-dos or team workflows |
| Microsoft Project | Gantt charts and resource scheduling | RoadmapAI does not require manual scheduling — it interprets data automatically |
| Tableau / PowerBI | Data visualisation and reporting | RoadmapAI does not require a data analyst to set up — it works from day one |

### 5.2 Core Design Principles

**1. Every view answers one decision**
Views are not organised by feature type (risks, finances, board) but by the question the user is trying to answer. Navigation = decision selection.

**2. Guardian is everywhere, not a destination**
AI output is injected into the view the user is already in. No extra clicks. No separate AI panel. The Decision Feed, financial recommendations, risk mitigations, and snapshot annotations are all contextual, inline, and immediately actionable.

**3. Numbers alone are not the product — interpretation is**
RoadmapAI does not display SPI. It says: "You are performing 23% below schedule pace — at this rate you will deliver 12 days late." The translation from metric to implication is the core value.

**4. Action > insight**
Every Guardian output includes a primary action button. The goal is not for the user to understand the situation — it is for the user to act. Approve, assign, dismiss, view — every decision item has a next step.

**5. Simplicity over completeness**
5 views instead of 7. A focused Execution view instead of a full Jira replacement. The product is opinionated: it does PMO intelligence exceptionally well and defers task management depth to existing tools.

### 5.3 Decision Feed — Core Differentiator

The Decision Feed is the feature that distinguishes RoadmapAI from all other PMO tools. It is not a list of alerts. It is a ranked list of decisions, each with:

- **Context:** why this matters right now
- **Consequence:** what happens if ignored (quantified)
- **Action:** what to do (one click)

Example feed for a critical project:

```
[CRITICAL]  Reduce scope of Sprint 4 by 2 features
            Current velocity cannot complete 14 features by Mar 31.
            Impact: saves 9 days, removes critical path risk.
            → [Apply scope reduction]   [View Sprint 4]

[WARNING]   External API dependency unowned — Sprint 4 at risk
            No owner assigned. Dependency unresolved for 8 days.
            Impact: will block 6 features if not resolved by Mar 15.
            → [Assign owner]   [Mark resolved]

[WARNING]   Budget forecast now $28k over BAC (CPI: 0.81)
            Resource cost running 19% above plan since Sprint 2.
            Impact: EAC = $148k vs budget $120k.
            → [View Financial Intelligence]   [Freeze resources]

[INFO]      Team at 112% capacity — burnout risk in Sprint 3
            3 resources above capacity threshold for 2+ weeks.
            → [View resource breakdown]
```

This feed is the primary reason an executive or PM opens RoadmapAI every morning.

| Risk mitigation suggestions | Risks & Dependencies | Claude Haiku | Per-risk, on demand | Structured mitigation options inline |
| Blocked feature suggestions | Execution | Claude (configurable) | Feature BLOCKED status | Unblocking action per blocked feature |

All AI model identifiers are configurable via the `ANTHROPIC_MODEL` environment variable (default: `claude-sonnet-4-5`).

## 8. Data Retention and Audit

- All user actions are logged to the `Activity` table with actor, timestamp, action type, and JSON metadata
- Project status changes are recorded in `ProjectStatusLog` (who changed, when, from, to, note)
- Snapshots provide point-in-time project state history with Guardian-annotated change impact
- Alerts are retained in the database with email delivery status
- No automatic data purge — data is retained indefinitely unless explicitly deleted

## 9. Limitations and Known Constraints

- **Decision Feed completeness:** The Decision Feed currently generates recommendations reactively (on feature change). Proactive daily generation on a schedule is not yet implemented — a cron-triggered regeneration would improve executive adoption
- **Guardian latency:** AI report generation takes 1–5 seconds. The Health & Decisions view loads the cached report immediately; stale state is shown until background refresh completes
- **Snapshot auto-trigger:** Requires `NEXT_PUBLIC_APP_URL` to be set correctly — internal `fetch()` call fails silently in misconfigured environments
- **Risk constraints:** Probability/impact validated (1–5) at API layer via Zod but no database-level check constraint
- **Status log actor:** `ProjectStatusLog.changedBy` stores a display name string, not a user ID — name changes do not retroactively update historical logs
- **Execution view depth:** The Board and Backlog are intentionally simple. Teams requiring deep sprint planning, sub-tasks, or workflow automation should integrate a dedicated tool (Jira, Linear) alongside RoadmapAI
- **Free plan:** Limited to 1 member — team collaboration requires Pro or Business tier
